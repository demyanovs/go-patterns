Garbage collector

В основе лежит алгоритм Mark and sweep.
Трехцветная алгоритм разметки. Алгоритм конкурентный, выполняется вместе с основной программой.
Алгоритм
Данные в куче рассматриваются как белые, серые и черные. Белые - не достижимые, серые - достижимые, черные - достигнутые.

Алгоритм состоит из двух фаз.
Первая фаза - фаза разметки. Все данные в куче помечаются как белые.
1. Помечаются все корни в серый цвет.
2. Выбираем помеченный серый объект из очереди м сканируем на наличие указателей.
3. Найденные объекты помечаются как серые, а исходный объект - как черный.
4. Повторяем пока очередь серых элементов не пуста.
5. Все белые данные удаляются.

Вторая фаза - фаза уборки. Удаляются все белые данные. Все серые данные становятся черными.
Все черные данные становятся белыми. Все черные данные, на которые указывают черные данные, становятся серыми. И так далее.
Пока есть черные данные.

Write barrier
* Трехцветный алгоритм разметки выполняется параллельно с основной программой, которая называется мутатором.
Для поддержания консистентности данных используется барьеры записи и чтения (write barrier).
* Одна из основных задача барьера записи - следить за тем, чтобы черные объекты не указывали на белые объекты.
* STW или stop the world - это фаза, когда программа полностью останавливается и выполняется только сборка мусора. Необходима для включения/выключения барьера записи.
Write barrier - это функция, которая вызывается при записи в память. Она помечает объекты, на которые указывает записываемый объект, как серые.

Когда запускать GC
Можно вручную вызвать запуск сборки мусора с помощью runtime.GC(). Однако в таком случае стоит быть осторожным,
так как это может привести к блокировке вызывамоей функции или всей прогламмы.
По умаолчанию GC запускается, когда размер кучи становится в 2 раза больше, чем размер кучи после предыдущего запуска GC.
Данный коэффициент можно изменить с помощью переменной GOGC.

Sweep termination !STW
Фаза завершения очистки. GC находится в состоянии _GCoff.
* Stop the world
* Ожидаем пока все goroutines достигнут safe point.
* Завершаем очистку ресурсов.

Mark phase
Фаза разметки. Для выполнения данной фазы выделяется примерно 25% CPU. GC находится в состоянии _GCmark.
* Включаем write barrier.
* Stop the world.
* Запускаем скаанирование глобальных переменных и стеков.
* При скаанировании стека работа горутины приостанавливается.
* Выполняем 3 цветный алгоритм для поиска мусора.

Mark termination !STW
Фаза завершения разметки. GC находится в состоянии _GCmarktermination.
* Stop the world.
* Дожидаемся завершения обработки последних задач из очереди.
* Очистка кэшей.
* STW не обязателен.
* Завершаем фазу разметки.

Sweep phase
Фаза очистки. Переходим в состояние _GCoff.
* Отключаем write barrier
* Start the world.
* Очистка ресурсов происходит в фоне.

Недостатки GC
* Не реализован алгоритм поколений
* Не реализовано уплотнение
* STW
* Нет возможности тонкой настройки

Оптимизации
GC может стать узким место в программе, когда речь идет о производительности.
Как мы выяснили, примерно 25% CP занимает фаза разметки и во время одглшл цикла раблты дважлы происходит STW.
Рекомендации:
* Уменьшить количество вызовов GC с помощью изменения GOGC.
* Использовать балласт.
* Использовать sync.Pool.
